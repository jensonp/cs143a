# Signals and Signal Handling

## Why This Topic Has to Appear

Once you understand processes, threads, interrupts, traps, and system calls, one question becomes unavoidable: **how does the operating system notify a process that something important has happened?** A process may need to be told that the user pressed `Ctrl-C`, that a child terminated, that a timer expired, that it attempted an illegal memory access, or that another process deliberately requested its attention. The process cannot poll for every such event without wasting CPU time, and the kernel cannot simply overwrite the process’s ordinary control flow without a disciplined mechanism. That mechanism is **signals**.

Signals matter because they are the process-level notification path for asynchronous events in Unix-like systems. They connect kernel-detected events and interprocess requests to user-visible process behavior. Without them, later topics such as child reaping, timer-driven behavior, terminal control, abnormal termination, and process supervision stay fragmented.

This topic therefore fills a specific gap: interrupts explain how hardware gets the CPU’s attention, traps explain how the current instruction stream enters the kernel, but signals explain how the kernel later informs a process that an event must be handled at the process level.

## The Object Being Introduced

A signal is not an ordinary function call, not a hardware interrupt, and not a thread by itself. It is a **kernel-mediated notification** attached to a process or thread of execution. Its job is to represent an event that matters to the target process and to force the question: what should happen to that process now?

What is fixed is that signals are OS-defined event notifications with specific meanings and default actions. What varies is:
- which signal was generated,
- what caused it,
- which process it is delivered to,
- whether the signal is blocked, pending, ignored, handled, or takes its default action,
- and what the target process was doing when the signal became relevant.

The conclusions signals allow are also structured. A signal may:
- terminate a process,
- stop it,
- continue it,
- notify it of a child state change,
- or invoke user-defined handling code.

So a signal is best understood as a process-facing event object with associated delivery and action rules.

## Formal Definition: Signal

A **signal** is an operating-system notification delivered to a process to indicate that a particular event has occurred and that some predefined or user-defined action must now be considered.

## Interpretation

In plain technical English, a signal is the kernel saying, “this process needs to know that something happened.” The first thing to notice is that a signal does not itself explain the whole response. It names the event class and then interacts with the process’s configured handling rules.

Signals are not random messages floating in the system. Each signal has a defined meaning and a default action. The process may sometimes replace the default action with a user-defined handler, but only within limits.

Lecture 2’s essential sequence is the right minimal model:
1. a signal is generated by some event,
2. the signal is delivered to a process,
3. the signal is handled either by its default action or a user-defined handler.

That three-step story is the backbone of the whole topic.

## Why Signals Exist

Signals solve a structural OS problem: some events matter to a process, but they do not arise as ordinary return values from the process’s current function call.

Examples:
- the user wants to interrupt a foreground program,
- a child process terminates,
- a timer expires,
- the process performs an invalid action,
- another process requests attention.

In each case, the event has to become visible to the target process even though the process may currently be computing, blocked in a system call, or otherwise not “asking” for the event at that moment. Signals provide that asynchronous notification path.

## Signal Generation

### Formal definition

**Signal generation** is the act by which the kernel determines that a particular signal should now exist for a target process because of some event or request.

### Interpretation

Generation means the operating system has recognized that some condition maps to a signal. The signal now conceptually exists as something that must later be dealt with according to the process’s signal state.

The first thing to notice is that generation is about **event recognition**, not immediate user-space handler execution. The event happens first; the signal object enters the kernel’s process-control logic second.

### Common causes of generation

Signals can be generated by several kinds of causes:

- **terminal or user actions**, such as interrupting a foreground job,
- **kernel-detected faults or exceptions**, such as invalid operations,
- **timers**, where elapsed time matters,
- **child state changes**, such as termination,
- **other processes**, which may send signals explicitly,
- **job-control events**, such as stop and continue requests.

This variety is important. Signals unify many different event sources into one process-facing mechanism.

## Signal Delivery

### Formal definition

**Signal delivery** is the point at which the operating system presents a generated signal to the target process so that the process’s configured action for that signal can take effect.

### Interpretation

Delivery is not the same thing as generation. A signal may be generated and then remain pending until the system reaches a point where it can be delivered according to the process’s state, masking rules, and scheduling situation.

The first thing to notice is that signals interact with execution state. The process may be:
- currently running,
- ready but not running,
- blocked in a system call,
- or stopped.

The kernel therefore needs rules for when the signal becomes active in the process’s control flow.

### What delivery checks conceptually

A serious mental model should separate these questions:

1. **Has a signal been generated for this process?**
2. **Is that signal currently blocked or deferred by the process’s signal mask?**
3. **If it is deliverable now, what action is configured for it?**
4. **Does delivery invoke a default action, mark the process state, or arrange execution of a handler?**

The answers determine what the OS is allowed to do next.

## Signal Handling

### Formal definition

**Signal handling** is the action taken once a delivered signal becomes effective for the target process, according to the signal’s default rule or a user-defined handler.

### Interpretation

Handling answers the question: **now that this signal is being delivered, what happens?**

Lecture 2 gives the essential split:
- **default handler**
- **user-defined handler**

That split is exactly right for an introductory OS model.

### Default handling

Each signal has a default action. Depending on the signal, the default may be to:
- terminate the process,
- stop the process,
- continue a stopped process,
- or simply notify in a defined way.

The key idea is that the kernel already knows what should happen if the process has not asked for custom behavior.

### User-defined handling

Some signals may instead be associated with a user-defined signal handler. In that case, when the signal is delivered, the process’s control flow is arranged so that the handler runs in user space under the operating system’s rules.

The important thing to notice is that this does **not** mean arbitrary privilege escalation. The handler still runs as part of the user process. The kernel is arranging process-visible control flow, not granting kernel authority.

### The Lecture 2 hard boundary

Two signals are especially important in the introductory model:
- `SIGKILL`
- `SIGSTOP`

Lecture 2 explicitly states that these **cannot be overridden**. That is a protection boundary, not a minor API detail. The operating system must retain certain non-negotiable control powers. A process is not allowed to install a user-defined handler that defeats those signals.

## Why `SIGKILL` and `SIGSTOP` Cannot Be Overridden

This deserves an explicit explanation.

If a process could override `SIGKILL`, then a misbehaving or malicious process could refuse termination completely.

If a process could override `SIGSTOP`, then the operating system and user environment would lose a critical control tool for suspending execution.

So the inability to override those signals is part of the larger theme of OS authority. User programs may customize some responses, but the kernel keeps final control over certain process-management actions.

## Signals vs Interrupts vs Traps

This distinction must be explicit.

### Signal vs interrupt

An **interrupt** is a hardware- or device-level event that gets the CPU’s attention and transfers control into the kernel.

A **signal** is a process-level notification used by the kernel to inform a process that some event matters to it.

So the rough relationship is:
- interrupts are CPU/kernel-facing,
- signals are process-facing.

A device interrupt may eventually contribute to a signal-like consequence for a process, but the mechanisms are not the same object.

### Signal vs trap

A **trap** is a synchronous kernel entry caused by the current instruction stream, such as a system call or exception.

A **signal** is the later process-level notification mechanism used to represent certain events or requests to a process.

A faulting instruction may trap into the kernel first, and the kernel may later decide that the process should observe the event as a signal. So again, traps and signals may be related in one causal chain, but they are not the same thing.

## Signals and Process State

Signals interact strongly with process state.

- A terminating signal may move a process toward termination.
- A stop signal may move a process into a stopped/suspended state.
- A continue signal may let a stopped process become runnable again.
- A child-state-change signal may matter to a parent that is blocked in or preparing for `wait`.

This is why signals belong conceptually near the process and lifecycle material, not only near interrupts. They are part of how the operating system manages living processes over time.

## A Fully Worked Example: Parent Notices Child Termination

This example is chosen because it connects signals to process lifecycle without dragging in too much API detail.

Suppose process P creates child process C. Later, C terminates.

### Step 1: the event occurs

The child reaches termination. That is a kernel-visible lifecycle event.

### Step 2: a signal is generated

The kernel determines that the parent process should be notified that a child state change occurred. So a signal corresponding to child-state notification is generated for P.

### Step 3: delivery becomes relevant

If P is running, the signal may become deliverable in its next signal-delivery opportunity according to the system’s rules. If P is blocked waiting in a child-related waiting mechanism, the kernel may use the event to wake or satisfy that wait. The signal and wait path are closely related conceptually: both are mechanisms by which the parent becomes aware of the child’s state change.

### Step 4: handling

If the parent uses default handling, the OS preserves the child’s status for later collection and the parent can call `wait` to retrieve it. If the parent has installed a handler for child-state notification, that handler may run and the process can decide what to do next.

### What this example teaches

The point is not just that “signals notify events.” The point is that signals are one of the OS’s process-facing coordination mechanisms. They turn kernel-recognized changes into process-visible control consequences.

## Another Worked Example: User Interrupts a Foreground Program

Suppose a user is running a foreground command in a terminal and presses an interrupt key combination.

### Step 1: user action occurs

The terminal subsystem recognizes the user’s interrupt request.

### Step 2: signal generation

The OS determines that the foreground process should receive the corresponding signal.

### Step 3: delivery

The signal becomes pending for that process and is delivered according to signal rules.

### Step 4: handling

If the process has not installed a user-defined handler for that signal, the default action applies. If it has installed an allowed user-defined handler, that handler runs instead.

### What this teaches

Signals are not just “errors.” They are also a control channel between the user environment, the kernel, and the process.

## Boundary Conditions and Failure Modes

A serious student should surface the following boundaries:

- **Generation is not delivery.** A signal may exist as pending before it is actually delivered.
- **Delivery is not arbitrary.** The process’s signal mask and current state matter.
- **Handling is constrained.** Not every signal may be overridden.
- **Default action is real behavior, not a placeholder.** If no user-defined handler exists, the default may terminate or stop the process.
- **Signals interact with waiting and blocking.** A blocked process is not outside signal logic.

A common failure in understanding is collapsing all three stages — generation, delivery, handling — into one instant. Doing so hides why masking, pending signals, and process state matter.

## Common Misconceptions

### Misconception 1: A signal is just an interrupt

False. Interrupts are hardware/kernel control transfers. Signals are process-level notifications.

### Misconception 2: A signal handler is like an ordinary callback with no OS meaning

False. Signal handling is part of the process-control model, with OS-defined entry conditions and constraints.

### Misconception 3: If a signal exists, the user-defined handler always runs immediately

False. The signal may be pending first, and masking/delivery rules matter.

### Misconception 4: All signals can be overridden

False. Lecture 2 explicitly identifies `SIGKILL` and `SIGSTOP` as non-overridable.

### Misconception 5: Signals matter only for killing processes

False. They also support notification, stop/continue control, timer-driven events, and child-state awareness.

Signals connect directly to:

They also reinforce a big OS theme: the system must preserve process isolation while still delivering meaningful events into process control flow.

## Conceptual Gaps and Dependencies

This topic assumes the reader already understands:
- what a process is,
- the difference between user mode and kernel mode,
- the difference between interrupts and traps,
- and the basic process lifecycle.

The most likely weak prerequisite is the distinction between **kernel-facing control events** and **process-facing notifications**. Students often know both words but not the direction in which each mechanism points.

This topic refers to several nearby concepts without fully teaching them:
- signal masks,
- pending signals,
- exact POSIX signal APIs,
- interaction with threads,
- interrupted system calls,
- and job-control/session details.

Those can be learned later. For Lecture 2 conversion, the essential backbone is generation, delivery, handling, default vs user-defined action, and the non-overridable signals.

## Retain / Do Not Confuse

### Retain
- A signal is a process-level OS notification that an event has occurred.
- The right minimal model is: **generated → delivered → handled**.
- Handling is either by a default action or a user-defined handler.
- `SIGKILL` and `SIGSTOP` cannot be overridden.
- Signals are process-facing, whereas interrupts are CPU/kernel-facing.

### Do Not Confuse
- Do not confuse a signal with an interrupt.
- Do not confuse signal delivery with signal generation.
- Do not confuse default handling with “no real action.”
- Do not confuse a signal handler with kernel-mode code.
- Do not confuse overridable signals with the kernel’s non-negotiable control signals.
